KMP Music 平台音频后端速查表

一眼判断：某个平台该用哪个播放引擎，哪些能力必须留在平台层，哪些能力可以抽象到 shared 层。

当前项目结论：Android 用 Media3；iOS 用 AVFoundation + AVAudioSession + MediaPlayer；当前 Desktop 是 JVM `jvm("desktop")`，所以 macOS DMG 和 Windows MSI 更适合先共用一个 JVM 桌面后端，再按需要升级到原生桥接。

推荐矩阵

平台 优先后端 系统控制 注意点
Android Media3 `ExoPlayer` `MediaSession` + `MediaSessionService` 处理音频焦点、通知、后台服务、耳机键和车机等 Android 生态能力。
iOS AVFoundation `AVPlayer` `MPNowPlayingInfoCenter` + `MPRemoteCommandCenter` `AVAudioSession` 设置 playback category；启用 Audio background mode；处理中断和路由变化。
macOS Desktop, 当前 Compose Desktop JVM 桌面后端：vlcj/LibVLC 或 GStreamer；MVP 可评估 JavaFX Media 需要额外接系统媒体键或托盘能力 当前 Gradle 是 `jvm("desktop")`，不是 Kotlin/Native macOS；直接用 AVFoundation 需要 JVM 到 Objective-C 桥接。
Windows Desktop, 当前 Compose Desktop JVM 桌面后端：vlcj/LibVLC 或 GStreamer；MVP 可评估 JavaFX Media 原生系统媒体控制通常需要 Windows API 桥接 WinUI `MediaPlayerElement` 和 Media Foundation 是原生 Windows 路线，不会自然融入 Compose Desktop JVM。

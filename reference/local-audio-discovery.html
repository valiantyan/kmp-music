KMP Music 本地音频发现速查表

把“扫描本地音乐”拆成授权、发现、元数据、可播放 URI、缓存同步五步。Android、iOS、Desktop 的差异主要在前两步。

总判断：可以实现跨平台本地音乐库，但 iOS 不能像 Android 一样全盘扫描手机音频。iOS 必须走用户显式授权的文件入口，或走系统音乐资料库 API；Desktop 则让用户选择文件夹后递归扫描。

平台路线

平台 P0 推荐 不推荐先做 输出给 common 的结果
Android `MediaStore.Audio` + 版本化读取权限。你熟悉的路线继续保留。 手写全盘文件遍历绕过 MediaStore。 `content://` URI、标题、歌手、专辑、时长、mime、size。
iOS 用户通过 Files / Document Picker 选择音频文件，App 导入到沙盒后扫描自己的目录。 假设能像 Android 一样扫描整个 iPhone 文件系统。 App 沙盒内 file URL、标题、歌手、专辑、时长、文件类型。
iOS 可选 `MPMediaLibrary` + `MPMediaQuery` 读取系统音乐资料库。 把 Apple Music / 云端 / DRM 内容都当成本地可播放文件。 媒体库 item id、元数据、可用时的 asset URL。
Desktop JVM 用户选择音乐文件夹，`Files.walk` 递归扫描，按扩展名、MIME 和解码能力过滤。 默认扫描用户全盘或系统目录。 本地 file path / URI、元数据、size、mtime、可读性。

iOS 三种来源

1. App 沙盒

App 自己下载、录制或导入的音频可以放在 Documents 或 Application Support，再用 `FileManager` 枚举。这个模型最稳定，也最适合 MVP。

2. Files 选择

用 `UIDocumentPickerViewController` 让用户从 Files、iCloud Drive 或第三方 document provider 选择文件。Import 是复制到沙盒；Open 是访问外部文件，通常要处理 security-scoped URL。

3. 系统音乐资料库

用 MediaPlayer 的 `MPMediaLibrary` / `MPMediaQuery`。需要隐私说明和授权；并且不是每个 item 都能给你可直接播放的本地文件 URL。

4. 不存在的路线

不要设计“扫描整个 iPhone 本地音频文件”的需求。iOS 的安全模型不提供这种通用文件系统访问。

字段建议

字段 说明 来源
`stableId` 跨启动稳定，不能只用列表下标。 Android media id；iOS imported file bookmark/hash；Desktop path + size + mtime hash。
`localUri` 给平台播放 engine 使用，不由 UI 拼接。 平台 scanner 输出，进入 common 时只当 opaque string 或 value class。
`durationMs` 进度条、排序和系统展示都需要毫秒时长。 Android MediaStore；iOS AVAsset / MPMediaItem；Desktop metadata parser 或播放库探测。
`sourceKind` 后续排查权限和播放失败时需要知道来源。 `androidMediaStore`、`iosImportedFile`、`iosMediaLibrary`、`desktopFolder`。

推荐 MVP 顺序

第一步：在 common 层把当前 `ScanLocalMusicUseCase` 从“状态切换 demo”演进为扫描编排接口，但不要接平台 API。

第二步：Android 接 `MediaStore.Audio`，iOS 接 Document Picker 导入音频，Desktop 接文件夹选择 + `Files.walk`。

第三步：把扫描结果映射到 `Song` / `PlayableMedia`，再交给播放 PRD 里的 `AudioPlayerEngine`。

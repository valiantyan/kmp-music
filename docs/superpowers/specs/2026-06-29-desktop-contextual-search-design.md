# macOS 桌面版上下文搜索设计

## 背景

当前 macOS 桌面版已经形成桌面 Shell：顶部标题栏、左侧主导航、页面工作区和底部播放器。搜索能力需要补齐，但不能把所有页面都简单塞进同一个“全局搜索框”模型里。

本设计基于已确认的信息架构：

- 搜索是二级页面，不是首页、收藏页或侧栏里的一个内嵌区块。
- 首页搜索本地扫描数据。
- 收藏页搜索收藏数据。
- 我的、设置、详情页和搜索页自身不显示顶部音乐搜索入口。
- 首页侧栏的输入框不是搜索页入口，而是本页局部筛选器。

## 目标

- 明确桌面端搜索入口出现在哪些页面。
- 明确不同入口进入搜索页后的数据范围、返回目标和页面文案。
- 避免顶部搜索框、页面内搜索框、首页侧栏筛选器三者语义混淆。
- 为后续 Compose 实现提供稳定的状态、导航和 UI 契约。

## 非目标

- 不引入在线音乐搜索。
- 不把设置页做成音乐搜索入口。
- 不重做首页、收藏、我的、设置的整体布局。
- 不把首页侧栏筛选器升级为搜索二级页入口。
- 不做持久化搜索结果缓存；搜索结果从当前数据实时派生。

## 核心结论

桌面端采用“上下文搜索”：

```text
首页顶部搜索 -> 搜索本地扫描数据
收藏顶部搜索 -> 搜索收藏数据
```

搜索页复用同一个页面组件，但由入口传入搜索上下文。

| 入口页面 | 顶部搜索是否显示 | 搜索范围 | 搜索页返回文案 | 搜索页副文案 |
| --- | --- | --- | --- | --- |
| 首页 | 显示 | 本地扫描出来的全部音乐数据 | `← 本地音乐` | `在本地音乐中搜索歌曲、专辑、歌手` |
| 收藏 | 显示 | 已收藏歌曲、专辑、歌手 | `← 收藏` | `在收藏中搜索歌曲、专辑、歌手` |
| 我的 | 不显示 | 无 | 无 | 无 |
| 设置 | 不显示 | 无 | 无 | 无 |
| 搜索页 | 不显示 | 当前搜索上下文 | 保持入口上下文 | 保持入口上下文 |
| 专辑/歌手/播放等二级页 | 不显示 | 无 | 无 | 无 |

## 信息架构

### 一级页面

Desktop 左侧 rail 保持：

- 首页
- 收藏
- 我的
- 设置

其中只有“首页”和“收藏”属于音乐内容集合页，所以显示顶部右上角音乐搜索入口。

### 二级搜索页

搜索页从首页或收藏进入，隐藏顶部右上角搜索框，只保留页面内主搜索框。

```text
首页
└── 搜索页（SearchContext.LocalLibrary）

收藏
└── 搜索页（SearchContext.Favorites）
```

搜索页返回时回到进入前页面，并保留对应左侧 rail 高亮：

- 从首页进入：左侧 `首页` 高亮，返回 `本地音乐`。
- 从收藏进入：左侧 `收藏` 高亮，返回 `收藏`。

## 顶部搜索入口

### 首页顶部搜索

首页顶部右上角保留搜索框，文案：

```text
搜索歌曲、专辑、歌手
```

交互：

1. 点击搜索框，进入搜索二级页。
2. 搜索页上下文为 `LocalLibrary`。
3. 顶部右上角搜索框在搜索页隐藏。
4. 页面内主搜索框自动聚焦。
5. 用户输入关键词后搜索本地扫描数据。

### 收藏顶部搜索

收藏页顶部右上角也保留搜索框，文案同首页：

```text
搜索歌曲、专辑、歌手
```

交互：

1. 点击搜索框，进入搜索二级页。
2. 搜索页上下文为 `Favorites`。
3. 顶部右上角搜索框在搜索页隐藏。
4. 页面内主搜索框自动聚焦。
5. 用户输入关键词后只搜索收藏数据。

### 我的与设置

我的和设置不显示顶部音乐搜索框。

原因：

- “我的”更偏账户、资料、入口聚合，不是直接音乐内容集合。
- “设置”是管理页面，放置“搜索歌曲、专辑、歌手”会造成语义冲突。
- 设置页未来如果需要搜索，应使用页面内部的“搜索设置项”，不能复用音乐搜索。

## 首页局部筛选器

首页本地音乐库侧栏保留一个弱化输入框，但它不是搜索页入口。

文案建议：

```text
筛选本地库
```

行为：

- 点击后只聚焦输入框。
- 输入后停留在首页，过滤当前本地音乐库列表。
- 不进入搜索二级页。
- 不显示搜索历史。
- 不影响顶部搜索页的查询状态。
- 清空后恢复当前本地库列表。

该输入框视觉应弱于顶部搜索入口：

- 更小高度。
- 更浅背景。
- 不使用强焦点边框。
- 文案使用“筛选”，避免“搜索”带来的跳转预期。

## 搜索页 UI

### 空查询状态

搜索页首屏为空查询/历史状态。

布局：

```text
← 本地音乐 / ← 收藏
搜索
在本地音乐中搜索歌曲、专辑、歌手 / 在收藏中搜索歌曲、专辑、歌手

[ 页面内主搜索框 ]

[ 全部 ] [ 歌曲 ] [ 专辑 ] [ 歌手 ]

最近搜索                    清空
[ 城市 ] [ 周杰伦 ] [ 孤勇者 ] [ Chinese Wedding ] [ 赵方婧 ]

输入关键词后显示匹配歌曲、专辑和歌手
```

规则：

- 页面内主搜索框是唯一可输入搜索框。
- 顶部标题栏右侧不显示搜索框。
- 不显示本地音乐库侧栏。
- 不显示右侧概览面板。
- 不显示“搜索建议”快捷入口。

### 有查询状态

输入关键词后展示结果。

结构：

```text
结果统计：找到 N 首歌曲、N 张专辑、N 位歌手
播放全部
排序：相关度 / 最近添加

歌曲结果表格
```

分段规则：

- `全部`：优先展示歌曲表格，必要时在下方展示少量专辑/歌手结果。
- `歌曲`：只展示歌曲表格。
- `专辑`：展示专辑结果。
- `歌手`：展示歌手结果。

歌曲表格列建议：

- `#`
- `标题`
- `歌手`
- `专辑`
- `时长`
- `来源/添加时间`
- 更多操作

### 空结果状态

当无匹配结果时：

```text
没有找到“{query}”相关内容
请尝试搜索歌曲名、专辑名或歌手名
```

收藏上下文可补充一句：

```text
当前只搜索已收藏内容
```

## 搜索历史

搜索历史需要做，但按上下文隔离。

```text
LocalLibrary 搜索历史
Favorites 搜索历史
```

记录时机：

- 用户按 Enter 执行搜索。
- 用户点击一个搜索结果。
- 用户从搜索结果打开专辑或歌手详情。

不记录：

- 只聚焦输入框。
- 只输入但未执行搜索。
- 首页侧栏 `筛选本地库` 的关键词。

历史规则：

- 去重，最新记录排最前。
- 每个上下文最多保留 8 到 10 条。
- 支持单条删除。
- 支持清空当前上下文历史。
- 后续如果增加云同步，搜索历史可作为用户偏好数据独立评估，不进入本次 MVP。

## 搜索结果与缓存

MVP 不做持久化搜索结果缓存。

原因：

- 搜索结果是当前本地库或收藏集合的派生数据。
- 本地扫描、收藏变更、元数据修正都会让缓存结果过期。
- 直接从当前状态实时派生更可靠。

可做轻量运行时优化：

- Controller 层保留当前 `query + scope + context + result`。
- 同一查询切换 `全部/歌曲/专辑/歌手` 时复用已派生结果。
- 本地库重新扫描或收藏集合变化后自动失效。

## 数据范围

### LocalLibrary

搜索范围：

- 当前本地扫描出来且可播放的歌曲。
- 从可播放歌曲聚合出来的专辑。
- 从可播放歌曲聚合出来的歌手。

不搜索：

- seed/mock 数据。
- 不可读、缺失、格式不支持的条目。
- 收藏之外的云端或在线内容。

### Favorites

搜索范围：

- 已收藏歌曲。
- 已收藏专辑。
- 已收藏歌手。

收藏项如果对应歌曲已不可播放，应在结果中明确不可用状态，不能静默替换为其他歌曲。

首版按“收藏实体”搜索：

- 用户收藏过的歌曲进入歌曲结果。
- 用户收藏过的专辑进入专辑结果。
- 用户收藏过的歌手进入歌手结果。
- 仅因某首收藏歌曲属于某专辑或某歌手，不自动把该专辑或歌手算作收藏专辑/歌手结果。

该规则避免收藏搜索把范围意外扩大，也和收藏页的歌曲/专辑/歌手分段语义一致。

## 状态模型建议

建议新增或扩展搜索上下文模型：

```kotlin
enum class SearchContext {
    LocalLibrary,
    Favorites,
}
```

搜索页面状态至少包含：

```text
context
query
scope
results
historyByContext
```

顶部搜索入口显示规则可以由当前页面推导：

```text
RootTab.Home -> show
RootTab.Favorites -> show
RootTab.Profile -> hide
SecondaryScreen.Settings -> hide
SecondaryScreen.Search -> hide
其他 SecondaryScreen -> hide
```

如果 Desktop 当前把设置作为 rail 常驻项，也按管理页处理：不显示音乐搜索。

- 收藏页顶部显示全局音乐搜索。
- 我的页、设置页不显示顶部音乐搜索。
- 从首页进入搜索页后，返回文案为 `← 本地音乐`。
- 从收藏进入搜索页后，返回文案为 `← 收藏`。
- 搜索页顶部标题栏右侧不再显示搜索框。
- 搜索页不显示本地音乐库侧栏、右侧概览面板、搜索建议快捷入口。

## 第一性原理审核

### 用户目标

用户搜索的真实目标不是“打开一个搜索页面”，而是在当前可信的数据集合里快速找到音乐内容。

因此入口必须回答两个基础问题：

- 我现在从哪里开始搜索？
- 搜索结果来自哪个数据集合？

本设计将搜索上下文绑定到入口页面：

- 首页代表本地音乐库，搜索本地扫描数据。
- 收藏代表用户主动保存的音乐资产，搜索收藏数据。

### 信息架构

搜索页是二级任务页面，不应该同时承担一级导航、局部筛选和全局入口三种角色。

因此：

- 顶部搜索框只负责进入搜索二级页。
- 搜索页只保留页面内主搜索框。
- 首页侧栏 `筛选本地库` 只做当前页过滤。

该分工避免两个搜索输入框同时出现，也避免局部筛选被误解成导航入口。

### 数据真实性

搜索结果必须从当前状态派生，不能从旧缓存或 mock 数据返回。

因此：

- LocalLibrary 搜索当前可播放本地扫描数据。
- Favorites 搜索当前收藏实体。
- 搜索结果不持久化缓存。
- 本地扫描或收藏变化后，搜索结果重新派生。

### 最小可用性

首版只保留用户能立即理解和使用的能力：

- 歌曲/专辑/歌手搜索。
- 当前上下文搜索历史。
- 空查询、空结果、有结果三种状态。

不引入在线搜索、设置页音乐搜索、跨上下文混合搜索和复杂推荐，避免 MVP 范围膨胀。

## 举一反三问题检查

### 从首页点击顶部搜索，搜索什么？

文档可回答。搜索上下文为 `LocalLibrary`，搜索本地扫描出来且可播放的歌曲、聚合专辑和聚合歌手。

### 从收藏点击顶部搜索，搜索什么？

文档可回答。搜索上下文为 `Favorites`，搜索已收藏歌曲、已收藏专辑和已收藏歌手。

### 在我的页能不能搜索歌曲？

文档可回答。我的页不显示顶部音乐搜索入口。

### 在设置页能不能搜索歌曲？

文档可回答。设置页不显示顶部音乐搜索入口；未来如需搜索，只能做页面内部的“搜索设置项”。

### 进入搜索页后，顶部右上角搜索框还在吗？

文档可回答。不显示，搜索页只保留页面内主搜索框。

### 首页侧栏 `筛选本地库` 会不会进入搜索页？

文档可回答。不会。它只在首页内过滤当前本地音乐库列表，不改变二级页面状态。

### 首页侧栏筛选词会不会进入搜索历史？

文档可回答。不会。搜索历史只记录搜索页中的执行搜索或结果点击行为。

### 收藏搜索历史和本地搜索历史是否混在一起？

文档可回答。不会。历史按 `LocalLibrary` 和 `Favorites` 上下文隔离。

### 收藏歌曲属于某个专辑，搜索收藏时这个专辑算收藏专辑结果吗？

文档可回答。不算。首版按收藏实体搜索，只有用户收藏过该专辑，才进入收藏专辑结果。

### 本地库重新扫描后，旧搜索结果还能继续展示吗？

文档可回答。搜索结果不做持久化缓存，扫描变化后从当前状态重新派生。

### 搜索页返回后回到哪里？

文档可回答。从首页进入返回 `本地音乐`，从收藏进入返回 `收藏`，并保留对应 rail 高亮。

### 切换左侧一级导航后搜索页是否保留？

文档可回答。测试建议要求切换 root tab 后清空或退出搜索二级页，避免上下文串页。

## 三轮交叉审核

### 第一轮：产品语义审核

- 问：顶部搜索框是否在所有页面都出现？
  答：否。只在首页和收藏出现，因为它们是音乐内容集合页。
- 问：搜索页是否还有两个搜索框？
  答：否。搜索页隐藏标题栏搜索，只保留页面内主搜索框。
- 问：首页侧栏输入是否会让用户误以为进入搜索页？
  答：通过文案 `筛选本地库` 和弱化视觉降低误解，并规定它不导航。
- 问：收藏搜索是否会让用户看到非收藏内容？
  答：不会。收藏上下文只搜索收藏实体。

### 第二轮：架构与状态审核

- 问：是否需要两个搜索页面？
  答：不需要。复用同一个搜索页组件，通过 `SearchContext` 切换数据范围、返回文案、副文案和历史记录。
- 问：顶部搜索显示规则应该写在页面里还是 Shell 里？
  答：应由 Desktop Shell 根据当前 root/secondary 状态统一推导，避免各页面重复实现。
- 问：搜索结果是否需要 Repository 级缓存？
  答：首版不需要。结果从当前曲库/收藏状态派生；Controller 可做轻量运行时复用。
- 问：扫描或收藏变更后如何避免脏结果？
  答：搜索结果不持久化；数据集合变化后重新计算。

### 第三轮：测试与验收审核

- 问：能否用测试证明首页和收藏进入了不同搜索上下文？
  答：可以。覆盖 `SearchContext.LocalLibrary` 与 `SearchContext.Favorites` 的进入路径。
- 问：能否证明我的/设置没有音乐搜索入口？

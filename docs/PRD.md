## 2. 产品概述

KMP Music 是一款基于 Kotlin Multiplatform 开发的本地音乐优先 App，面向需要稳定播放、本地曲库管理、本地收藏、歌手与专辑浏览、个性化设置的用户。产品优先完成完整本地音乐播放闭环：扫描本地音乐、浏览音乐、播放音乐、收藏音乐、管理个人偏好。

首个版本以“本地优先、轻量、清晰、可跨平台复用”为核心目标，不追求复杂社交、短视频、直播、电台、在线下载等大型音乐平台能力。产品应先建立可靠的信息架构、播放体验和跨端一致的 UI 与状态管理，并预留登录与云同步能力：MVP 默认游客模式完整可用，同时提供登录入口，为后续收藏、播放历史、设置等数据云同步打基础。

## 3. 产品目标

### 3.1 业务目标

1. 建立 Android、iOS 与 Desktop 可复用的音乐 App 基础框架。
2. 完成首页、我的、播放、收藏、歌手、专辑、设置等核心页面。
3. 支持本地音乐扫描、播放、暂停、切歌、播放列表、收藏、歌手与专辑浏览。
4. 首版支持游客模式和登录入口，为后续登录、云同步、推荐系统预留产品与技术空间。

### 3.2 用户目标

1. 用户可以快速找到本地曲库里的音乐。
2. 用户可以稳定地播放音乐并控制播放状态。
3. 用户可以收藏歌曲、歌手和专辑，形成个人音乐库。
4. 用户可以查看歌曲所属歌手、专辑信息。
5. 用户可以按个人习惯调整播放、主题、本地扫描、缓存等设置。

### 3.3 技术目标

1. 业务逻辑、数据模型、播放状态、仓库层尽量复用在 shared 模块。
2. Android、iOS 与 Desktop 保持一致的信息架构和核心交互。
3. UI 组件尽量通过 Compose Multiplatform 复用。
4. 播放能力根据平台封装差异接口，避免平台代码污染业务层。

## 4. 目标用户

### 4.1 核心用户

| 用户类型 | 特征 | 核心需求 |
| --- | --- | --- |
| 普通听歌用户 | 高频打开 App 播放歌曲 | 快速播放、收藏、查看最近播放 |
| 专辑型用户 | 偏好完整专辑收听 | 浏览专辑、按专辑顺序播放 |
| 歌手型用户 | 关注特定歌手 | 查看歌手热门歌曲、专辑列表 |
| 本地音乐用户 | 拥有本地音乐文件 | 扫描本地音乐、离线播放 |

### 4.2 使用场景

1. 通勤时打开 App，从首页继续播放最近歌曲。
2. 用户听到喜欢的歌曲后点击收藏，之后从收藏页面快速找到。
3. 用户在播放页点击歌手名，进入歌手页查看其他歌曲。
4. 用户在歌手页进入专辑页，按专辑曲序播放。
5. 用户在设置页切换深色模式、重新扫描本地音乐或清理缓存。

## 5. 产品范围

### 5.1 MVP 范围

| 模块 | MVP 是否包含 | 说明 |
| --- | --- | --- |
| 首页 | 包含 | 本地曲库概览、最近播放、常听歌曲、专辑入口 |
| 我的页面 | 包含 | 用户音乐资产入口、最近播放、本地音乐入口、登录入口 |
| 播放页面 | 包含 | 播放控制、进度、播放队列入口 |
| 收藏页面 | 包含 | 收藏歌曲、收藏专辑、收藏歌手 |
| 歌手页面 | 包含 | 歌手信息、热门歌曲、专辑列表 |
| 专辑页面 | 包含 | 专辑信息、歌曲列表、播放全部 |
| 设置页面 | 包含 | 主题、播放、缓存、关于 |
| 搜索 | 建议包含基础版 | 搜索歌曲、歌手、专辑 |
| 登录 | 包含入口 | 首版游客模式完整可用，提供登录入口和登录页占位 |
| 云同步 | 预留 | 首版不强制同步，后续同步收藏、播放历史、设置 |
| 在线支付/会员 | 暂不包含 | 后续商业化再定义 |
| 社交评论 | 暂不包含 | 非 MVP 核心闭环 |
| 在线歌曲下载 | 暂不包含 | 首版不支持下载在线歌曲到本地 |

### 5.2 非目标范围

1. 首版不做复杂社区、评论、动态、直播。
2. 首版不做版权采购、会员购买、广告投放。
3. 首版不强制登录，游客模式可以使用播放、收藏、最近播放、设置等核心能力。
4. 首版不支持下载在线歌曲到本地。
5. 首版歌词不作为必须功能，不在 MVP 验收范围内。
6. 首版推荐基于本地曲库、最近播放、收藏偏好实现，不要求算法推荐。

## 6. 信息架构

### 6.1 主导航

App 在 Android 与 iOS 使用底部 Tab 作为主导航，Desktop 使用左侧导航栏。三端保持同一套信息结构：

1. 首页
2. 收藏
3. 我的

播放页、歌手页、专辑页、设置页通过页面内入口进入，不固定在移动端底部导航中。Desktop 可在左侧导航中常驻设置入口，但页面层级与移动端保持一致。

### 6.2 页面层级

```text
App
├── 首页
│   ├── 搜索页
│   ├── 播放页
│   ├── 歌手页
│   └── 专辑页
├── 收藏
│   ├── 收藏歌曲
│   ├── 收藏歌手
│   ├── 收藏专辑
│   ├── 播放页
│   ├── 歌手页
│   └── 专辑页
├── 我的
│   ├── 最近播放
│   ├── 本地音乐
│   ├── 登录入口
│   └── 设置
└── 全局迷你播放器
    └── 播放页
```

## 7. 核心功能需求

### 7.1 首页

#### 页面目标

首页负责承接用户打开 App 后的第一步行为，优先展示本地曲库中可立即播放的音乐内容，并提供搜索、最近播放、常听歌曲、专辑和歌手等入口。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 搜索入口 | P0 | 点击进入搜索页 |
| 最近播放 | P0 | 展示最近播放歌曲，支持继续播放 |
| 常听歌曲 | P0 | 基于本地播放历史展示歌曲卡片或列表 |
| 本地专辑 | P1 | 展示专辑封面、名称、歌手 |
| 本地歌手 | P1 | 展示歌手头像、名称 |
| 重新扫描入口 | P1 | 本地曲库为空或需要刷新时进入扫描流程 |

#### 关键交互

1. 点击歌曲后立即加入当前播放队列并开始播放。
2. 点击专辑进入专辑页面。
3. 点击歌手进入歌手页面。
4. 点击迷你播放器进入播放页面。

#### 验收标准

1. 首页首屏必须展示至少一个可播放内容区；如果未扫描到音乐，展示本地扫描入口。
2. 首页内容为空时展示空状态，不出现白屏。
3. 切换底部 Tab 后，首页滚动位置应尽量保留。

### 7.2 我的页面

#### 页面目标

我的页面集中展示用户个人音乐资产与 App 管理入口，包括最近播放、本地音乐、登录入口、设置等。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 用户信息区域 | P0 | 游客状态展示默认头像、昵称和登录入口 |
| 登录入口 | P0 | 首版进入登录页占位，后续承接云同步 |
| 最近播放入口 | P0 | 进入最近播放列表 |
| 本地音乐入口 | P0 | 扫描并展示本地歌曲 |
| 收藏入口 | P1 | 可跳转收藏页 |
| 设置入口 | P0 | 进入设置页面 |

#### 关键交互

1. 点击本地音乐后展示本地歌曲列表。
2. 点击设置进入设置页。
3. 游客状态点击用户信息进入登录页占位，说明登录后可用于云同步收藏、播放历史和设置。

#### 验收标准

1. 我的页面入口清晰，不依赖登录也能使用核心能力。
2. 本地音乐无权限时应提示用户授权。
3. 未扫描到本地音乐时展示空状态和重新扫描入口。

### 7.3 播放页面

#### 页面目标

播放页面是 App 的核心体验，提供清晰的播放状态、歌曲信息、进度控制、播放模式、收藏和播放队列。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 歌曲封面展示 | P0 | 展示专辑封面或默认封面 |
| 歌曲信息 | P0 | 歌名、歌手、专辑 |
| 播放/暂停 | P0 | 控制当前歌曲 |
| 上一首/下一首 | P0 | 切换播放队列 |
| 进度条拖动 | P0 | 支持 seek |
| 收藏歌曲 | P0 | 收藏或取消收藏 |
| 播放模式 | P1 | 顺序、单曲循环、随机 |
| 播放队列 | P1 | 展示当前队列 |
| 后台播放 | P0 | 锁屏或切后台后继续播放 |

#### 关键交互

1. 点击播放按钮后按钮状态立即反馈。
2. 拖动进度条时暂停进度自动刷新，松手后跳转到目标时间。
3. 点击歌手名进入歌手页面。
4. 点击专辑名进入专辑页面。
5. 点击收藏图标后状态应即时更新，并同步到收藏页。

#### 验收标准

1. 播放、暂停、切歌状态在首页迷你播放器和播放页面保持一致。
2. 当前歌曲不存在时展示默认空播放状态。
3. 播放失败时展示错误提示，并允许重试或跳过。
4. App 切到后台后，播放状态不应丢失。

### 7.4 收藏页面

#### 页面目标

收藏页面用于聚合用户主动标记的歌曲、歌手与专辑，帮助用户快速回到喜欢的内容。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 收藏歌曲列表 | P0 | 默认 Tab |
| 收藏歌手列表 | P1 | 展示歌手卡片 |
| 收藏专辑列表 | P1 | 展示专辑卡片 |
| 取消收藏 | P0 | 支持从列表直接操作 |
| 批量播放 | P1 | 播放全部收藏歌曲 |
| 空状态 | P0 | 引导用户去首页发现音乐 |

#### 关键交互

1. 点击收藏歌曲后开始播放。
2. 点击收藏歌手进入歌手页。
3. 点击收藏专辑进入专辑页。
4. 取消收藏后当前列表即时刷新。

#### 验收标准

1. 收藏状态在所有页面保持一致。
2. 收藏列表为空时展示明确空状态。
3. 取消收藏不能误删本地音乐或历史记录。

### 7.5 歌手页面

#### 页面目标

歌手页面展示歌手基本信息、热门歌曲、专辑列表，为用户提供按歌手维度探索音乐的路径。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 歌手头图/头像 | P1 | 可使用默认图 |
| 歌手名称 | P0 | 必须展示 |
| 收藏歌手 | P1 | 收藏或取消收藏 |
| 热门歌曲 | P0 | 歌曲列表，支持播放 |
| 专辑列表 | P0 | 点击进入专辑页 |
| 歌手简介 | P2 | 可折叠展示 |

#### 关键交互

1. 点击热门歌曲后从该歌手歌曲列表开始播放。
2. 点击播放全部后按热门歌曲顺序播放。
3. 点击专辑进入专辑页面。
4. 点击收藏按钮更新收藏状态。

#### 验收标准

1. 歌手不存在或数据为空时展示错误/空状态。
2. 歌手页歌曲播放后，播放队列应与歌手歌曲列表一致。
3. 收藏歌手后收藏页的歌手 Tab 应出现该歌手。

### 7.6 专辑页面

#### 页面目标

专辑页面展示专辑基本信息和曲目列表，支持按专辑顺序播放与收藏专辑。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 专辑封面 | P0 | 展示封面或默认封面 |
| 专辑名称 | P0 | 必须展示 |
| 歌手信息 | P0 | 点击进入歌手页 |
| 发行信息 | P2 | 年份、曲目数 |
| 收藏专辑 | P1 | 收藏或取消收藏 |
| 曲目列表 | P0 | 展示歌曲序号、歌名、时长 |
| 播放全部 | P0 | 按曲目顺序播放 |

#### 关键交互

1. 点击播放全部后，将专辑全部歌曲加入播放队列。
2. 点击单曲后从当前曲目开始播放，并保留专辑队列。
3. 点击歌手名进入歌手页面。
4. 点击收藏按钮同步收藏页。

#### 验收标准

1. 专辑曲目顺序必须稳定。
2. 专辑无曲目时展示空状态。
3. 播放队列中应保留专辑上下文，方便上一首/下一首。

### 7.7 设置页面

#### 页面目标

设置页面提供播放行为、外观、缓存、隐私和应用信息配置。

#### 功能列表

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 主题设置 | P0 | 跟随系统、浅色、深色 |
| 播放设置 | P1 | 自动播放、播放模式默认值 |
| 桌面端窗口设置 | P2 | Desktop 端可记录窗口尺寸和布局偏好 |
| 缓存管理 | P1 | 查看缓存大小、清理缓存 |
| 本地音乐扫描 | P0 | 重新扫描本地音乐 |
| 语言设置 | P2 | 后续国际化 |
| 关于 App | P0 | 版本号、开源协议 |
| 隐私政策 | P1 | 后续发布前补齐 |

#### 关键交互

1. 切换主题后全局立即生效。
2. 清理缓存前需要二次确认。
3. 点击重新扫描本地音乐后展示扫描进度。

#### 验收标准

1. 设置项变更后应持久化。
2. 清理缓存不能删除用户收藏和播放历史。
3. 设置页面在无网络状态下仍可正常打开。

## 8. 全局功能需求

### 8.1 迷你播放器

迷你播放器在首页、收藏、我的等主页面底部展示，位于底部导航上方。

功能要求：

1. 展示当前歌曲封面、歌名、歌手。
2. 支持播放/暂停。
3. 支持点击进入播放页面。
4. 当前无播放歌曲时不展示或展示轻量空状态。

### 8.2 搜索

基础搜索优先搜索本地曲库，支持歌曲、歌手、专辑三类结果。

功能要求：

1. 输入关键词后展示搜索结果。
2. 搜索结果按歌曲、歌手、专辑分组或通过 Tab 切换。
3. 点击歌曲播放，点击歌手/专辑进入详情页。
4. 搜索为空时展示历史搜索、本地曲库统计或扫描入口。

### 8.3 播放队列

功能要求：

1. 记录当前播放列表来源：首页、收藏、歌手、专辑、本地音乐。
2. 支持上一首、下一首。
3. 支持顺序播放、单曲循环、随机播放。
4. 当前播放歌曲在队列中高亮。
5. 队列入口参考主流音乐产品的通用模式：移动端从播放页或迷你播放器打开底部弹层，Desktop 可打开右侧队列面板。
6. 队列内支持播放指定歌曲、移除单曲、拖拽调整顺序、清空队列。
7. 执行清空队列、移除正在播放歌曲等高影响操作时，需要提供二次确认或可撤销提示。
8. 队列变更后，迷你播放器、播放页、列表高亮状态必须同步更新。

### 8.4 本地音乐

功能要求：

1. 支持音频格式：mp3、flac、aac、wav、m4a。
2. Android 请求音频文件读取权限，授权通过后扫描当前手机可播放音乐。
3. iOS 请求媒体库或文件访问权限，授权通过后扫描当前手机可播放音乐。
4. Desktop 由用户指定本地目录，App 导入该目录下合法且受支持的音频文件。
5. 扫描歌曲元数据：歌曲名、歌手、专辑、时长、封面、音频格式。
6. 本地音乐应支持离线播放。
7. 扫描结果应持久化，并支持手动重新扫描。
8. 不支持或无法读取的音频文件应跳过，并在扫描结果中展示失败数量或失败原因入口。

### 8.5 账号与云同步

MVP 默认游客模式完整可用，但保留登录与云同步的产品入口。

功能要求：

1. 我的页面展示登录入口。
2. 登录页首版可以是占位页，说明登录后可同步收藏、播放历史和设置。
3. 游客模式下收藏、播放历史、设置保存在本地。
4. 后续接入账号后，应支持将本地数据合并到云端账号。
5. 未登录状态不得阻塞本地音乐扫描、播放、收藏、搜索等核心流程。
6. 登录方式、同步字段、冲突合并策略暂不作为 MVP 详细设计内容，后续版本单独补充。

## 10. 非功能需求

### 10.1 性能

1. Android 与 iOS 冷启动到首页可交互目标小于 2 秒，Desktop 启动到主窗口可交互目标小于 3 秒。
2. 播放点击反馈目标小于 300ms。
3. 列表滚动应保持流畅，避免主线程执行重 IO。
4. 封面图片加载应有占位、缓存和失败兜底。

### 10.2 稳定性

1. 播放失败不能导致 App 崩溃。
2. 网络异常、本地文件不存在、权限拒绝均需要明确状态。
3. App 进入后台、锁屏、恢复前台后播放状态保持一致。

### 10.3 跨平台一致性

1. Android、iOS 与 Desktop 页面结构一致。
2. 核心业务状态由 shared 层统一管理。
3. 平台差异通过 expect/actual 或平台适配器隔离。
4. 移动端优先底部导航，Desktop 优先侧边导航，但路由和页面模型保持一致。

### 10.4 UI 设计风格

1. 整体风格偏系统原生，遵循平台用户熟悉的导航、返回、弹窗、列表和设置交互。
2. 视觉表达保持年轻潮流：封面、播放页、重点按钮和动效可以更有活力，但不牺牲可读性和操作效率。
3. Android、iOS、Desktop 保持统一品牌感，同时允许平台控件细节自然贴近各自系统。
4. 颜色、圆角、字体层级需要形成设计令牌，便于 Compose Multiplatform 复用。

### 10.5 Desktop 布局与快捷键

Desktop 版本参考主流音乐产品的常见桌面布局模式，优先保证高频播放操作稳定可达。

1. Desktop 使用左侧导航栏，常驻首页、收藏、我的、设置等主入口。
2. 主内容区根据窗口宽度响应式展示列表、专辑网格、歌手网格和详情页。
3. 播放控制区固定在窗口底部，展示当前歌曲、播放控制、进度条、音量和队列入口。
4. 播放队列优先使用右侧面板承载，避免遮挡主内容。
5. 支持记忆窗口尺寸和布局偏好。
6. 建议支持基础快捷键：Space 播放/暂停，上一首/下一首快捷键，搜索快捷键，音量增减快捷键。
7. 具体快捷键组合需兼容目标桌面系统规范，避免与系统全局快捷键冲突。

### 10.6 可访问性

1. 播放、收藏、返回等核心按钮应提供语义描述。
2. 文本颜色对比度满足基础可读性。
3. 重要操作区域点击范围不小于 44x44 dp/pt。

### 10.7 隐私与权限

1. 本地音乐扫描前说明权限用途。
2. 不上传用户本地音乐文件。
3. 收藏、播放历史等本地数据应允许用户清理。

## 12. 状态与异常

| 场景 | 产品表现 |
| --- | --- |
| 首页无数据 | 展示空状态和本地音乐扫描入口 |
| 收藏为空 | 引导去首页发现音乐 |
| 本地音乐无权限 | 展示权限说明和授权按钮 |
| 桌面端未选择音乐目录 | 展示选择目录入口 |
| 扫描到不支持格式 | 跳过文件，并在扫描结果中展示失败数量或失败原因 |

## 13. MVP 里程碑

### 阶段一：基础框架

1. 搭建 KMP 工程结构。
2. 建立导航、主题、基础组件。
3. 定义 Song、Artist、Album、PlaybackState 数据模型。
4. 建立 shared 层 Repository 与 UseCase 基础结构。

### 阶段二：核心页面

1. 实现首页、我的、收藏页面。
2. 实现播放页面和迷你播放器。
3. 实现歌手页面和专辑页面。
4. 实现设置页面。

### 阶段三：播放闭环

1. 接入平台播放能力。
2. 实现播放队列和播放模式。
3. 实现收藏、最近播放、本地持久化。
4. 实现本地音乐扫描。

### 阶段四：验收与优化

1. 补齐空状态、错误状态、加载状态。
2. 完成 Android、iOS 与 Desktop 主要流程验证。
3. 优化启动、滚动、播放响应性能。
4. 准备首版发布清单。

## 14. 验收主流程

1. 用户打开 App，首页展示本地曲库内容和最近播放。
2. 用户点击一首歌曲，迷你播放器出现并开始播放。
3. 用户进入播放页面，可以暂停、继续、切歌、拖动进度。
4. 用户收藏当前歌曲，收藏页面出现该歌曲。
5. 用户从歌曲进入歌手页，再进入专辑页。
6. 用户在专辑页点击播放全部，播放队列按专辑曲序更新。
7. 用户进入我的页面，打开本地音乐并完成扫描。
8. 用户进入我的页面，点击登录入口，可以进入登录页占位，返回后游客模式继续可用。
9. 用户进入设置页面，切换深色模式并返回首页，主题保持生效。

## 15. 已确认产品决策

1. 首版音乐来源以本地音乐为主。
2. 产品需要登录与云同步能力，MVP 采用游客模式完整可用加登录入口的方式落地。
3. UI 设计风格偏系统原生，同时保持年轻潮流的视觉表达。
4. 歌词不作为 MVP 必须功能。
5. 首版不支持下载在线歌曲到本地。
6. 首版目标平台为 Android、iOS、Desktop。
7. 首版本地音乐支持 mp3、flac、aac、wav、m4a。
8. Android 与 iOS 授权后扫描当前手机可播放音乐，Desktop 由用户指定目录导入合法音频文件。
9. 登录方式、云同步字段、冲突合并策略可后续补充，不阻塞 MVP。
10. 首页排序规则可后续细化，不阻塞 MVP。
11. 播放队列交互与 Desktop 布局优先参考主流音乐产品的成熟模式。

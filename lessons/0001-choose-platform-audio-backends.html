Lesson 0001

选择 KMP 音乐播放器的各平台音频后端

这一课只解决一个问题：在当前 KMP Music 里，Android 已经知道用 Media3，那么 iOS、macOS Desktop、Windows Desktop 应该怎么选，并且怎样不污染 shared 层。

一句话答案：iOS 走 Apple 原生 AVFoundation；当前 Desktop 目标是 JVM Compose Desktop，所以 macOS DMG 和 Windows MSI 先按 JVM 桌面应用选后端，优先评估 vlcj/LibVLC 或 GStreamer，MVP 可用 JavaFX Media 做低风险试验。

先分清两个问题

播放引擎

负责打开文件、解码、播放、暂停、seek、进度、结束、错误。Android 的答案是 Media3 `ExoPlayer`；iOS 的同类答案是 AVFoundation `AVPlayer`。

系统整合

负责后台播放、锁屏信息、耳机键、系统媒体键、音频焦点、打断恢复。它通常不是同一个 API，所以不要把它塞进 shared UI。

平台选择

平台 推荐 为什么
Android Media3 `ExoPlayer` + `MediaSessionService` Media3 的 `Player` 是统一控制接口，`ExoPlayer` 是默认实现；`MediaSessionService` 用于把播放器和会话放在独立服务中支持后台播放。见 Media3 简介 和 Media3 ExoPlayer。
iOS AVFoundation `AVPlayer` + `AVAudioSession` + MediaPlayer AVFoundation 是 Apple 时间媒体框架，覆盖 iOS 和 macOS；`AVAudioSession` 的 playback category 是媒体播放 App 的基础；`MPNowPlayingInfoCenter` 和 `MPRemoteCommandCenter` 负责锁屏信息与远程控制。见 AVFoundation overview、Configuring Audio Settings。
macOS Desktop 当前项目先按 JVM Desktop 选：vlcj/LibVLC 或 GStreamer；若未来做原生 macOS 后端，再看 AVFoundation 你的 Gradle 是 `jvm("desktop")`，打 DMG 只是 JVM 桌面包，不是 Kotlin/Native macOS App。直接调用 AVFoundation 会需要 JVM 到 Objective-C 桥接，复杂度高于先接一个 JVM 桌面播放库。
Windows Desktop 当前项目先按 JVM Desktop 选：vlcj/LibVLC 或 GStreamer；Windows 原生路线是 Media Foundation / WinUI MediaPlayer Microsoft 的 Media Foundation 是 Windows 原生媒体平台，WinUI `MediaPlayerElement` 有内建 transport controls 和系统媒体控制整合；但它们不是 Compose Desktop JVM 的自然依赖，需要桥接或单独 Windows UI。见 Media Foundation 和 Media players。

对当前项目的架构形状

不要让页面知道 `ExoPlayer`、`AVPlayer` 或 LibVLC。页面只发“播放这首歌、暂停、seek 到这里”的意图；控制器或 use case 更新共享状态；平台 engine 执行真实播放，并把进度、结束、错误事件回流。

Desktop 后端怎么取舍

JavaFX Media

最快验证，API 简单，适合 MVP spike。缺点是格式和系统整合能力有限，后续可能换引擎。

vlcj / LibVLC

对本地音频格式更稳，跨 Windows、macOS、Linux。缺点是要处理原生 LibVLC 分发和 vlcj 授权。

GStreamer

插件和 pipeline 能力强，适合高级音频处理。缺点是学习和打包复杂度较高。

原生桥接

macOS 用 AVFoundation，Windows 用 Media Foundation。系统融合最好，但在当前 JVM Desktop 架构里工程量最大。

练习：先回忆，再展开

1. iOS 后台播放只靠 `AVPlayer` 够吗？

先用自己的话回答，再点开参考答案。

不够。`AVPlayer` 是播放引擎；后台播放还需要正确的 `AVAudioSession` category 和 Background Modes。锁屏信息与远程控制再交给 MediaPlayer 框架。

2. 当前项目的 macOS Desktop 是否天然应该用 AVFoundation？

先根据 `composeApp/build.gradle.kts` 判断。

不一定。当前 Desktop 是 `jvm("desktop")`，macOS DMG 仍是 JVM Compose Desktop 应用。AVFoundation 是 Apple 原生路线，直接接入需要桥接，所以先评估 JVM 桌面后端更贴近当前架构。

3. shared 层最应该抽象什么？

只说最小集合。

抽象命令、状态和事件：load/play/pause/seek/stop，当前进度、播放状态、结束、错误。不要抽象平台专属细节，比如 Android notification 或 iOS remote command handler。
